01 / Overview

02 / Why memory matters

As conversations stretch across sessions, tools, and devices, retrieving relevant facts becomes compute-heavy and unreliable. A larger window helps, but it does not guarantee correct recall, low cost, or stable long-term state.

Memory is an external, non-parametric layer around an LLM system. It decides what to preserve, how to organize it, when to retrieve it, and what to forget. That makes it qualitatively different from simply appending more tokens to a prompt.

For future agent systems, memory also cannot remain private to one chat. Multiple agents and workflows may need a shared view of the user or task—while still resolving conflicts and keeping local information appropriately scoped.

02.1 / Seven failure modes

Eleven recurring issues in the literature collapse into seven broader system failures. Fixing only retrieval leaves most of the pipeline untouched.

  1. 01

    Information density

    Raw history contains filler, repetition, and low-value context. More tokens can create less signal.

    SimpleMem · LightMem
  2. 02

    Retrieval efficiency

    Broad searches over large memory stores increase latency, cost, and irrelevant recall.

    SwiftMem
  3. 03

    Temporal modeling

    Dialogue order is mistaken for event time; persistent and evolving states are flattened into points.

    TSM · TiMem
  4. 04

    Single granularity

    One memory level cannot serve fact lookup, summaries, patterns, and complex multi-hop questions.

    MemGAS · TiMem
  5. 05

    Disconnected evidence

    Facts stored as isolated atoms fail when questions require causal or relational reasoning.

    AriadneMem
  6. 06

    Update and consolidation

    Changing user state creates stale facts, contradictions, and tension between online and offline writes.

    LightMem · AriadneMem · TiMem
  7. 07

    Shared memory and orchestration

    Multi-agent systems must decide what stays local, what becomes global, who can write, and how conflicting updates are resolved.

03 / What recent papers fix

No single design solves agent memory. Each system works on a different stage: what to write, how to consolidate, how to index, how to represent time, or how to connect evidence.

04 / Comparative analysis

These papers are best understood not as competing storage formats, but as optimizers for different stages of an end-to-end memory system.

SimpleMem and LightMem reduce noisy memory construction. SwiftMem improves retrieval efficiency. TiMem and TSM make time explicit. MemGAS adapts memory granularity. AriadneMem adds relational structure for connected evidence.

05 / Cost-aware adaptive memory

Future systems should choose a memory strategy dynamically, based on the query, task complexity, state history, and cost budget.

08 / Conclusion

Long context alone will not solve agent memory.

Memory must optimize the full pipeline—not just storage or retrieval. For production agents, the central challenge is to preserve the right information in the right form, keep it connected and current, and do so without wasting compute.

That makes memory a systems problem: part database, part retrieval engine, part temporal model, and part coordination layer.
